# MetricDB 具体方向汇报

## 负责方向

根据项目需求文档，我当前负责的是第 4 个子项目：`From Vector Indexing to Metric Indexing`，即把传统的向量索引能力扩展为适用于 MetricDB 的度量索引能力。对应说明见 [PROJECT_REQUIREMENTS.md](./PROJECT_REQUIREMENTS.md)。

这个方向的核心不是做 SQL 解析，也不是做 anchor points 的发现，而是在跨数据库、跨 embedding 模型的数据已经能够被映射到统一 metric 视图之后，解决“如何高效建索引、如何高效检索”的问题。

## 方向背景

MetricDB 的总体目标是构建一个不依赖单一 embedding 模型的数据库系统，使不同向量数据库中的数据能够在统一的 metric 视角下进行查询。传统向量数据库通常绑定某个具体 embedding 模型，因此不同数据库之间难以直接比较和联合检索。

在这个背景下，`metric indexing` 的作用是把“理论上可比较”进一步推进到“系统上可高效查询”。没有索引层支撑，跨库检索虽然可能成立，但在真实数据规模下很难具有实用性。

## 这部分要解决的核心问题

### 1. 统一度量空间下如何建索引

多个异构向量数据库在完成对齐、映射或整合之后，需要一个统一的索引结构来支撑近邻搜索。这里的问题不只是存储数据，而是要让查询具有较好的延迟、吞吐和扩展性。

### 2. 能否复用已有向量索引

实际系统中索引构建成本很高。如果每次整合不同数据库后都从零构建索引，代价可能过大。因此还需要研究能否将一个数据库中已有的索引结构进行转换，并与另一个数据库中的索引融合。

## 文档中给出的两条技术路线

项目文档给出了两条主线：

### 路线一：整合向量后，从零构建 metric index

先把来自不同数据库的 embedding 向量整合到统一的 metric 视图，然后基于整合后的结果重建索引。

这条路线的优点是系统结构更清晰、逻辑更统一，也更适合先做出一个可运行原型。缺点是重建代价较高，并且比较依赖前面映射与整合结果的质量。

### 路线二：转换已有索引并进行索引合并

不是先把所有数据重建索引，而是尝试把现有数据库中的索引结构变换到新的 metric 视图中，或把多个索引直接合并。

这条路线更有研究性，也更偏系统优化，但实现难度更高。因为不仅要考虑向量表示如何变化，还要考虑原有索引的结构性质在变化后是否仍然有效。

## 我具体要做的工作

### 1. 文献与方法调研

梳理常见向量索引方法，分析它们依赖的前提条件，以及这些前提在 metric database 场景下哪些仍然成立，哪些会失效。

### 2. 问题建模

明确输入、输出和评价目标：

- 输入：来自不同数据库的 embedding、数据库间映射关系、可能已有的局部索引。
- 输出：支持统一近邻检索的 metric index。
- 目标：在准确性和效率之间取得平衡。

### 3. 实现主线原型

更务实的主线应当是优先实现“整合后重建索引”的方案。因为这一方案更容易形成完整系统，也更适合作为毕业项目主体成果。

### 4. 尝试扩展方案

如果主线跑通，再继续研究“索引转换与合并”的可行性，把它作为增强模块或进阶研究点。

### 5. Benchmark 与评估

这一部分不仅要证明“能查”，还要证明“查得快”。因此实验至少需要覆盖：

- 检索准确性或召回效果
- 索引构建时间
- 查询延迟
- 吞吐量
- 内存或存储开销

## 模块边界

### 不属于我主要负责的部分

- Metric SQL parser 和 logical optimiser
- Predicate vector search 的语法与改写规则
- Anchor points 或 reference mappings 的发现算法
- Soft union / hard union 的总体查询策略设计

### 属于我主要负责的部分

- 统一 metric 视图上的索引设计
- 索引的构建、转换与合并
- 检索效率优化
- 模块实现、实验和 benchmark

## 汇报时可以直接使用的表述

我负责的是 MetricDB 项目中的 `From Vector Indexing to Metric Indexing` 子方向。这个方向的重点不是去解决向量如何对齐，而是在跨 embedding 数据已经能够被整合之后，进一步解决如何建立统一、高效的索引结构，从而支持实际可用的近邻检索。根据项目需求，我的工作主要沿两条路线展开：一是对整合后的向量重新构建 metric index；二是研究如何将已有 vector index 进行转换和合并。整体上，我的重点会放在索引结构设计、系统实现和 benchmark 评测上，使 MetricDB 不只在概念上成立，也在系统性能上具备可用性。

## 建议的执行顺序

1. 先做“整合后重建索引”的可运行原型。
2. 明确实验数据、索引方法和评价指标。
3. 在主线稳定后，再推进“索引转换与合并”的扩展研究。

这样更容易尽快形成可演示、可评估、可写进 proposal 或 progress report 的结果。
